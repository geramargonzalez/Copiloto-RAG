# Proposito
Definir cuando y como escalar consultas que el copiloto no puede resolver con certeza. El piloto busca respuestas confiables con evidencia; por eso, ante falta de información suficiente se debe priorizar claridad y derivación rápida al equipo correcto.

# Cuándo escalar
Escalar cuando falte ID de referencia, haya conflicto entre fuentes o la consulta requiera acceso a sistemas no conectados en modo read-only. Escalar también cuando el usuario solicite datos personales o acciones de escritura no permitidas por el alcance del copiloto.

# Pasos de escalamiento
1. Indicar que no hay certeza con la información disponible.
2. Pedir 1 a 3 datos mínimos estrictamente necesarios para continuar.
3. Informar el sistema o documento recomendado para verificar (portal financiero, SOP de devoluciones, tablero de pedidos).
4. Crear ticket interno con resumen del caso, requestId y área destino.
5. Confirmar al usuario tiempos estimados de respuesta del canal humano.

# Riesgos
Escalar tarde puede aumentar retrabajo y frustración del usuario. Escalar sin contexto técnico mínimo genera tickets incompletos y demoras. El equilibrio recomendado es pedir datos concretos y derivar de inmediato cuando la evidencia sea insuficiente.

# Evidencia y registro
Toda derivación debe registrar pregunta original, limitación detectada y fuentes consultadas. Esta trazabilidad permite mejorar cobertura documental del copiloto y reducir futuros casos sin respuesta.
